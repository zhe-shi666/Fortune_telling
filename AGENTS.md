# Autonomous App Builder

本文件用于驱动 AI 在当前项目中进行“低人工介入”的 0 到 1 iOS App 开发。

它不是写给人提问用的，而是写给 AI 自主推进时遵守的执行规则。

当前项目仅保留这一份总控文件作为单一事实来源；`CLAUDE.md` 只作为跳转引用，不再单独维护。

## 1. 固定输入

每次启动时，优先读取以下文件：

- `docs/rules.md`
- `docs/app.md`
- `docs/domain.md`
- `docs/api.md`
- `skills/页面生成助手/SKILL.md`
- `skills/逻辑实现助手/SKILL.md`
- `skills/测试助手/SKILL.md`

如果这些文件已经足够，不再额外请求大而全的补充文档。

## 2. 自主启动规则

如果用户只给出一句话描述，也允许启动。

启动顺序如下：

1. 先判断 `docs/app.md` 是否仍是占位模板。
2. 若仍是占位模板，根据用户的一句话描述补齐 `docs/app.md` 的 v0 版本。
3. 再判断 `docs/domain.md` 是否仍是占位模板。
4. 若仍是占位模板，根据 `docs/app.md` 补齐 `docs/domain.md` 的 v0 版本。
5. 再判断 `docs/api.md` 是否仍是占位模板。
6. 若仍是占位模板，根据 `docs/app.md`、`docs/domain.md` 和页面语义补齐 `docs/api.md` 的 v0 版本。
7. 只有在 `app.md`、`domain.md` 和 `api.md` 达到最小可用状态后，才进入页面生成阶段。

当信息不足时：

- 优先生成最保守的 MVP
- 显式写出假设
- 不自动扩张成复杂多模块产品

## 3. 阶段识别

AI 每次启动时，按以下顺序判断当前处于哪个阶段：

### 阶段 A：文档初始化

如果满足任一条件，进入该阶段：

- `docs/app.md` 仍大量保留“待补充”
- `docs/domain.md` 仍大量保留“待补充”
- `docs/api.md` 仍大量保留“待补充”
- 用户只提供了一个产品想法，还没有可用文档

该阶段目标：

- 自动补全 `docs/app.md` v0
- 自动补全 `docs/domain.md` v0
- 自动补全 `docs/api.md` v0

### 阶段 B：页面生成

如果文档已可用，但核心页面还没有第一页骨架，进入该阶段。

该阶段目标：

- 选择 MVP 中最关键的第一个页面
- 使用页面生成助手产出页面骨架、契约、mock 和 `ui-layout-report.md`
- 如果项目尚无可运行入口，则在首个页面骨架生成后，立即装配最小可运行 app 壳

可运行 app 壳至少包含：

- 最小 app 入口
- 根视图或根路由
- 首个 MVP 页面装配
- 用于人工审核的最小 mock 或占位数据

页面优先级判断规则：

1. 优先选择 `docs/app.md` 页面清单中排列最靠前的页面。
2. 如果关键用户流程中第一步明确出现了某个页面，则该页面优先级高于普通列表顺序。
3. 如果页面清单顺序与关键流程顺序冲突，优先采用“关键用户流程第一步出现的页面”。
4. 如果仍无法判断，则选择最能形成 MVP 主闭环的入口页，而不是设置页、营销页或次级详情页。

### 阶段 C：逻辑实现

如果页面骨架已达到交接条件，但逻辑层尚未成型，进入该阶段。

该阶段目标：

- 使用逻辑实现助手补全 ViewModel、业务流和 `logic-report.md`

### 阶段 D：测试与质量

如果页面和逻辑已达到测试交接条件，但 QA 尚未建立，进入该阶段。

该阶段目标：

- 使用测试助手生成最小测试矩阵、测试代码或测试计划，以及 `QA-Dashboard.md`

### 阶段 E：Feature 收尾

如果当前 feature 的页面、逻辑、测试都已完成，则：

- 回看报告中的阻塞项
- 如果没有严重阻塞，则推进下一个 MVP feature
- 如果有严重阻塞，则优先回写 `app.md` / `domain.md` / `api.md` / 报告，而不是盲目生成更多代码

## 4. 交接判断

AI 不得凭感觉切换阶段，必须遵守以下条件。

### 页面生成助手完成条件

只有满足以下条件之一，才能切换到逻辑实现阶段：

- 已产出最小页面契约、页面骨架和 `ui-layout-report.md`
- 或已在 `ui-layout-report.md` 中明确写出阻塞项与降级方案

检查点：

- 若项目从 0 开始，当前页面已能通过最小运行入口被打开审核
- 页面包含最小 `State` / `Action` 骨架
- 场景覆盖至少考虑 `ideal`、`loading`、`empty`、`error`
- 报告包含 `confidence`、`complexAdaptation`、关键假设

### 逻辑实现助手完成条件

只有满足以下条件之一，才能切换到测试阶段：

- 已产出 compile-safe 逻辑骨架和 `logic-report.md`
- 或已在 `logic-report.md` 中明确写出关键 Stub、事务步骤和失败注入点

检查点：

- 高影响动作已有处理策略
- `logic-report.md` 包含 `Critical Stubs`
- 若适用，包含 `Transaction Steps`
- 若适用，包含 `Failure Injection Points`

### 测试助手完成条件

只有满足以下条件之一，才能视为当前 feature 测试阶段完成：

- 已生成最小可运行测试
- 或已生成测试计划与完整 `QA-Dashboard.md`

检查点：

- 有测试矩阵
- 有 `Skipped`
- 有 `Warnings`
- 有 `readinessScore` 或 `pending`

## 5. 自主循环规则

AI 每轮最多推进一个 feature，不同时铺开多个 feature。

每轮循环遵守：

1. 读文档
2. 判断阶段
3. 先完成当前阶段的最小骨架或报告
4. 如果当前是首个 MVP 页面且尚无运行入口，则在页面骨架完成后补齐最小运行壳
5. 执行当前阶段剩余最小任务
6. 判断是否达到交接条件
7. 若达到，则进入下一阶段
8. 若未达到，则优先补文档、报告或骨架

如果本轮实现、测试或设计解析反向暴露出新的产品范围、业务规则、接口契约或交接判断变化，必须先做变更回流，再进入下一阶段。回流顺序固定为：

1. `docs/app.md`
2. `docs/domain.md`
3. `docs/api.md`
4. 当前阶段报告
5. skill

不要只修改代码或报告而放任上游文档继续漂移。

整体循环终止条件：

- 当 `docs/app.md` 页面清单中的所有 MVP 页面都已完成页面、逻辑、测试三个阶段时，停止自主循环。
- 停止时输出一份整体完成摘要，至少包含：已完成页面、仍存在的阻塞项、当前 `advisory` / `blocking` 状态、建议的下一步。
- 输出摘要后等待人工确认是否进入下一轮方向决策，而不是自动扩展到新的 MVP 外功能。

## 6. 何时停止自动推进

只有在以下情况才允许停止并请求人工确认：

- 用户的一句话描述存在多个高风险解释，且会导致完全不同的产品方向
- 涉及支付、医疗、法律、合规等高风险领域，当前文档又严重不足
- 关键业务规则互相矛盾，无法通过最保守假设收敛

除了这些情况，优先继续推进最小可用版本。

## 7. 优先修改哪里

若当前结果不理想，优先按以下顺序修正：

1. `docs/app.md`
2. `docs/domain.md`
3. `docs/api.md`
4. 当前阶段报告
5. skill

不要一遇到偏差就先重写 skill。

## 8. 总原则

- 优先保守，不优先炫技
- 优先闭环，不优先铺量
- 优先可迭代，不优先伪完整
- 优先显式假设，不优先脑补
