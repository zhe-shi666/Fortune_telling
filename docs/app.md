# App

这个文件只描述当前 app，不写跨项目通用规则。

## 1. 一句话目标

- 帮助对东方命理感兴趣的用户，用一份命主档案快速查看今日运势，并完成八字测算、取名、合婚与充值入口的 MVP 闭环。

## 2. 目标用户

- 核心用户：希望通过移动端快速查看每日运势、命理测算和名字建议的中文用户。
- 常见场景：早晨打开 `今日页面` 查看当日运势；录入出生信息后切到 `八字`、`取名`、`合婚` 页面做延伸测算；在需要更多高级能力时进入 `充值页面`。

## 3. MVP 范围

当前版本只做最小可用闭环，按页面划分如下：

- `今日页面`：展示今日运势娱乐参考、宜忌、签语预览，支持打开解签面板、进入命主档案、进入充值页面。
- `八字页面`：录入出生日期/时辰/性别/历法，点击 `合取测算` 后消耗 1 灵玉并展示娱乐参考摘要；若已开通 VIP，则不扣减灵玉。
- `取名页面`：输入姓氏、性别、出生日期与出生时辰后点击生成名字建议，每次点击生成消耗 1 灵玉；首次显示 2 个，同输入再次点击继续追加；更换姓氏、性别、日期或时辰时清空当前推荐，并按输入性别约束名字风格与排序，并支持打开收藏清单弹层。
- `合婚页面`：录入双方信息后点击 `合婚推演`，每次推演消耗 1 灵玉；若已开通 VIP，则不扣减灵玉。
- `充值页面`：展示灵玉余额、VIP 权益状态、内购选项、支付方式与返回按钮；当前点击购买按钮统一展示克制的购买说明文案，后续计划接入真实内购。
- `命主页面`：查看并保存命主档案，保存后用于今日、八字、取名、合婚的后续推演。

## 4. 当前不做

- 账号体系、登录注册、多端同步。
- 真实支付 SDK、订单回调、发票、退款、合规结算。
- 社区分享、社交关系、消息推送。
- 复杂后台管理、运营位配置、A/B 实验平台。
- 复杂命理公式校验与专家人工服务，当前先保留为娱乐参考文案和保守结果。

## 6. 关键用户流程

### 流程 A：今日运势主闭环

1. 用户进入 `Fortune / Daily / TodayOverview`
2. 若本地无命主档案，页面降级为引导态，用户点击 `档` 进入 `Profile / MasterProfile / ProfileEditor`
3. 用户保存出生信息后返回 `TodayOverview`
4. 系统展示今日干支、宜忌与签语摘要
5. 用户点击 `解签`，系统先校验权益；普通用户消耗 1 灵玉，VIP 则直接打开解签面板

### 流程 B：命主档案驱动测算

1. 用户进入 `TodayOverview` 或直接进入 `BaziAnalysis`
2. 系统读取本地命主档案，若字段不足则提示补齐日期/时辰/性别/历法
3. 用户完成输入并触发测算
4. 系统返回最小结果卡片或 mock 结果
5. 用户继续切换到 `NamingWorkshop` 或 `CompatibilityReading`

### 流程 C：充值入口闭环

1. 用户从 `TodayOverview` 点击 `充` 进入 `RechargeCenter`
2. 用户选择一个充值方案与支付方式
3. 当前 MVP 仅保留内购入口与方案展示，点击购买按钮统一展示购买说明文案
4. 用户点击返回回到 `TodayOverview`

## 7. App 特有约束

### 页面特有

- 页面阶段优先以 Pencil MCP 中的 6 张移动端设计稿为 UI 起点；系统状态栏、设备外框、Home Indicator 不转为业务组件。
- `今日页面` 是首个 MVP 页面，必须优先保证它可被运行和人工审核。
- 底部主导航只包含 `今日 / 八字 / 合婚 / 取名` 四个入口；`充值` 与 `命主档案` 通过页面内按钮进入。
- 设计稿中未明确的复杂交互统一降级为最小可编译骨架，并在对应报告中记录 `confidence` 与 `complexAdaptation`。

### 逻辑特有

- 命主档案采用本地优先策略；在真实后端未锁定前，不阻塞页面与逻辑骨架交付。
- 今日、八字、取名、合婚结果当前允许使用保守推断结果，但必须统一表述为娱乐参考或辅助参考，不能伪装成真实命理结论。
- 充值页面当前只保留未来内购的页面骨架与提示文案，不接真实支付链路；但灵玉余额与 VIP 权益状态需要先在本地可读。
- 解签、合取测算、取名生成、合婚推演统一遵守同一条使用规则：普通用户每次消耗 1 灵玉，VIP 无限使用。

### 测试特有

- `TodayOverview` 当前门禁为 `advisory`，优先覆盖页面状态切换和主交互。
- 真实支付相关内容一律不进入当前轮次自动化测试，只在文档和 QA 报告中明确跳过。
- 对 Pencil 未清晰定义的区域，测试以结构和状态断言为主，不做高置信度快照承诺。

## 8. 当前版本成功标准

- 用户可以打开 app，直接看到 `TodayOverview` 首屏并进行人工审核。
- 用户可以通过 `档` 进入 `ProfileEditor`，保存后回流刷新今日、八字、取名与合婚页面。
- 用户可以通过底部导航切换 `今日 / 八字 / 合婚 / 取名` 四个主页面，并在每页看到最小可运行的 mock 结果或保守降级态。
- 用户可以从 `TodayOverview` 进入 `RechargeCenter`，完成方案和支付方式选择，并在点击购买按钮后看到克制的购买说明提示。
- 六个 MVP 页面都至少覆盖 `ideal`、`loading`、`empty`、`error` 中与当前页面相关的场景，并附带页面、逻辑、测试三类产物与阶段报告。

## 9. 当前关键假设

- UI 结构来自 Pencil MCP 当前已打开的设计树，而不是截图推断。
- 今日运势、八字、合婚与取名页面已切换到项目内的传统命理算法底座，并统一表述为娱乐参考或辅助参考，不伪装成专业命理结论。
- 取名页面当前显式收集姓氏、性别、出生日期与出生时辰；取名算法不再依赖午时回落假设，并需把性别作为候选约束条件之一。
- 算法底座当前已支持公历、普通农历与农历闰月输入；但仍需继续补黄金样例校对，不能把当前结果包装成流派级权威结论。
- 传统命理级算法的目标规格已单独定义在 [算法规格-v1.md](/Users/linx/Documents/lth/Fortune_telling/docs/算法规格-v1.md)；后续实现应以该规格为准，而不是继续扩张模板型规则。
- 充值页面虽然存在，但本轮不进入真实支付、合规或财务能力实现；当前仅保留未来内购方向的 UI 骨架与购买说明文案，相关接口仍保留最小假设契约。
- 设计中使用的 `Playfair Display`、`Newsreader` 等字体在 iOS 工程中暂以系统 serif 风格近似，不额外引入字体资源。
- `NamingWorkshop` 的收藏清单当前会保存在设备本地 `SwiftData` 仓储中，不做多端同步或账号绑定。
- `CompatibilityReading` 默认以当前命主档案作为一方输入，另一方使用保守样例值做演示，不将演示结果包装成真实婚配建议。
