# 商业化 TODO 清单

这个清单基于当前项目现状整理：

- 当前命理结果已统一为娱乐参考 / 辅助参考方向
- 当前灵玉 / VIP 为本地状态，不是线上权益系统
- 当前充值页仍是占位说明，不是真实内购
- 当前档案、收藏、权益主要保存在本地

目标不是“立刻提审”，而是把项目推进到“具备商业化落地条件”的状态。

## 0. 已确认产品决策

以下内容已由产品方向确认，不再作为待讨论项：

- 商业模型采用 `灵玉 + VIP 并行`
- 四个消耗动作统一规则：
  - 普通用户每次 `解签 / 合取测算 / 取名 / 合婚推演` 消耗 `1` 灵玉
  - `VIP` 用户上述四类动作均不消耗灵玉
- `VIP` 权益边界仅为“免除灵玉消耗”，当前不包含额外专属内容
- 灵玉为永久有效，不做过期清零
- 充值页目标版本为真实商业页，不再长期停留在“还未开放”占位状态
- 真实商业页文案保持克制，不夸大权益，不制造误导性承诺

建议统一对外文案口径：

- `解签 / 合取测算 / 取名 / 合婚推演，每次消耗 1 灵玉；开通 VIP 后可不限次使用。`

## 3. 权益与订单后端

这部分是商业版里最关键的“后台能力”。严格说，不一定一上来就要大而全的数据后端，但商业版至少建议有权益后端。

### 3.1 必做后端

- [ ] 建立权益后端：记录用户当前灵玉余额、VIP 状态、最近更新时间
- [ ] 建立订单校验能力：校验内购结果并生成可信权益状态
- [ ] 建立消费流水：记录每次解签 / 测算 / 取名 / 合婚的灵玉扣减
- [ ] 建立幂等机制：避免重复发货、重复扣减、重复恢复
- [ ] 建立异常对账能力：购买成功但未到账时可补发

### 3.2 建议后端

- [ ] 建立匿名用户体系或设备级用户 ID
- [ ] 建立档案同步能力，让换机后权益与资料可恢复
- [ ] 建立收藏清单同步能力
- [ ] 建立服务端配置能力：商品价格文案、活动开关、功能开关
- [ ] 建立风控与限流策略，避免接口被刷

## 11. 一句话结论

如果你要做商业化版本，严格来说不是只差“两个后端”。

更准确地说，你至少还差这 4 块：

- [ ] 真实内购能力
- [ ] 权益 / 订单后端
- [ ] 正式合规文档
- [ ] 商业化体验与测试收口

其中最小商业化闭环建议是：

1. `StoreKit 2 + 商品配置`
2. `权益后端 + 订单校验`
3. `灵玉 / VIP 与四个功能页联动`
4. `隐私政策 / 用户协议 / 提审说明`
