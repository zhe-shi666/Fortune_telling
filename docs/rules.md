# Rules

这个文件是项目级通用规则，优先级高于单次任务描述。适用于当前项目，也可以复用到大部分中小型 iOS App。

## 1. 目标

- 保持开发规则稳定，让三个 skill 的协作方式始终可预测。
- 保持技术边界稳定，避免每次任务都重新猜架构。
- 优先生成可编译、可测试、可降级的代码，而不是一次性“看起来很完整”的代码。
- 优先尽快交付可运行、可人工审核的 app 壳，而不是先堆完整文档。
- 保持总控入口唯一，避免 `AGENTS.md` 与其他总控文件重复维护后漂移。

## 2. 开发输入文件

从 0 开发时，优先依赖以下输入，而不是先设计项目路径：

- `skills/页面生成助手/SKILL.md`
- `skills/逻辑实现助手/SKILL.md`
- `skills/测试助手/SKILL.md`
- `docs/rules.md`
- `docs/app.md`
- `docs/domain.md`
- `docs/api.md`

如果这 7 个文件的信息仍不够，再补当前任务需要的最少上下文，不要先扩成大而全的文档体系。

## 3. 技术边界

- 平台：默认 iOS 17+。
- 语言：Swift 6+。
- 并发：默认开启严格并发检查。
- UI：优先 `SwiftUI`，除非系统能力不足，不主动引入 `UIKit` 包装。
- 架构：默认 `MVVM + Coordinator`。
- 持久化：优先 `SwiftData`。
- 网络：优先 `URLSession + Codable`。
- 依赖管理：优先 `Swift Package Manager`。
- 第三方库：默认从严，只有明确收益时才引入。

补充约束：

- 默认遵循 Swift API 设计指南。
- 网络请求优先使用 `Endpoint` 枚举或等价的集中契约。
- 若使用 `SwiftData`，模型关系和并发边界必须显式可读。

## 4. 代码约束

- View 保持尽量无状态，只消费 `State`，通过闭包或 `send(_:)` 派发动作。
- ViewModel 负责状态变化与业务流程，不直接持有页面布局逻辑。
- ViewModel 不导入 `UIKit`，也不承载页面布局实现。
- 不在 View 中硬编码导航目标。
- 复杂页面优先拆组件，不输出超长单文件。
- 异步流程优先使用 `async/await`。
- 未确认存在的基础设施不要脑补，例如自定义容器、事务 DSL、日志框架。
- 如果项目从 0 开始且缺少运行入口，优先生成最小可运行入口、根路由和首屏装配，让 app 能被打开审核。
- 将 Pencil 设计中的设备外框、系统状态栏、Home Indicator 视为系统 UI 或设计容器，不转成业务页面代码。
- 状态栏时间、电量、信号、Wi-Fi、刘海、灵动岛、底部手势条默认只作为安全区和布局参考，不生成业务组件。
- 只有当设计明确要求自绘状态栏或设备展示效果时，才允许保留这类元素，并必须标记为装饰性 UI。

## 5. 三个 Skill 的协作顺序

默认按这个顺序工作：

1. 页面生成助手
2. 逻辑实现助手
3. 测试助手

反向迭代时，优先读取已经存在的页面报告、逻辑报告和测试报告；如果这些报告不存在，再根据代码与文档自行补全判断。

## 6. 共享协议

三个 skill 统一使用以下字段：

- `MockScenario`：`ideal`、`loading`、`empty`、`error`
- `confidence`：`0.0 ... 1.0`
- `complexAdaptation`：布尔值
- `criticality`：`high`、`medium`、`low`
- `gateMode`：`advisory`、`blocking`
- `transactionSteps`：数组，每项至少包含 `id`、`name`、`criticality`、`rollbackRequired`

这些字段是协作协议。字段名需要稳定，文件路径以各 skill 的输出合同为准。

`confidence` 不只用于描述，还必须驱动降级策略。统一阈值如下：

- `0.85 ... 1.0`：高置信度。允许按设计正常落地结构、交互与测试，只需记录少量假设。
- `0.60 ... 0.84`：中置信度。允许实现主结构与标准交互；复杂动效、复合组件和非关键细节必须降级，并记录假设。
- `0.40 ... 0.59`：低置信度。只实现最小可编译骨架、核心动作和语义占位；`complexAdaptation` 默认设为 `true`；逻辑与测试同步降级。
- `< 0.40`：极低置信度。不得脑补复杂 UI 或副作用；优先输出占位、阻塞项或人工确认点。若该区域承载关键路径且无安全降级方案，可把当前 feature 标为 `blocking`。

三个 skill 在处理同一页面时，必须消费同一套阈值，而不是各自凭感觉解释 `confidence`。

## 7. 文档职责

- `docs/rules.md`：项目通用规则，只写可跨 app 复用的边界和协议。
- `docs/app.md`：当前 app 的目标、范围、页面与流程。
- `docs/domain.md`：当前 app 的实体、状态和业务规则。
- `docs/api.md`：当前 app 的后端接口契约和错误语义。

如果某条规则只属于当前 app，不要写进 `rules.md`，写进 `app.md`、`domain.md` 或 `api.md`。

## 8. 从 0 开发的最小流程

1. 先在 `docs/app.md` 写清 app 目标、MVP 页面和关键流程。
2. 再在 `docs/domain.md` 写清核心实体、状态和高影响动作。
3. 再在 `docs/api.md` 写清 MVP 需要的最小接口契约。
4. 用页面生成助手生成第一批页面骨架，并选出首个 MVP 页面。
5. 在首个页面骨架生成后，立即装配最小可运行 app 壳：运行入口、根路由、首个 MVP 页面装配和 mock 驱动。
6. 用逻辑实现助手补页面契约和业务流。
7. 用测试助手补最小测试矩阵和 QA 报告。
8. 在每轮迭代里优先修正文档和 skill 暴露出的歧义，不先扩张目录。

## 9. 无人工启动流程

当目标是“从一句话描述启动 0 到 1 开发”时，允许 AI 先自动补齐 `app.md`、`domain.md` 和 `api.md` 的 v0 版本，再进入代码阶段。

最小输入只需要一句话，至少包含以下信息中的 2 项：

- 这个 app 帮谁用
- 这个 app 解决什么问题
- 用户第一版要完成什么动作

自动启动规则如下：

1. 如果 `docs/app.md` 仍是占位内容，先根据一句话描述补成 v0。
2. 如果 `docs/domain.md` 仍是占位内容，基于 `app.md` 自动补成 v0。
3. 如果 `docs/api.md` 仍是占位内容，基于 `app.md`、`domain.md` 和页面语义自动补成 v0。
4. 所有自动补全的内容必须显式保留假设，不允许伪装成已确认事实。
5. 若一句话描述不足以区分多个高风险业务方向，只允许收敛到最保守的 MVP，不自动扩张功能。

`app.md` 的 v0 至少要补到以下程度：

- 一句话目标
- 目标用户
- MVP 范围
- 当前不做
- 页面清单
- 至少一个关键流程

`domain.md` 的 v0 至少要补到以下程度：

- 1 到 3 个核心实体
- 关键字段
- 状态枚举
- 高影响动作
- 关键业务规则
- 已知风险

`api.md` 的 v0 至少要补到以下程度：

- MVP 需要的最小接口清单
- 每个接口的 `Method` / `Path` / 用途
- 最小 request / response 字段
- 基本错误语义
- 已知假设与风险

## 10. Skill 交接条件

只有满足当前阶段的最小完成标准，才能交给下一个 skill。

### 页面生成助手 -> 逻辑实现助手

满足以下条件之一即可交接：

- 已生成最小页面契约、页面骨架和 `ui-layout-report.md`
- 或者已经明确列出阻塞项，并在 `ui-layout-report.md` 中写清原因

最小完成标准：

- 若项目从 0 开始且此前没有运行入口，已生成最小可运行 app 壳，使当前页面可以被打开审核
- 页面存在可继续迭代的 `State` / `Action` 骨架
- 至少覆盖 `ideal`、`loading`、`empty`、`error` 四个标准场景中的相关项
- `ui-layout-report.md` 已写入 `confidence`、`complexAdaptation`、关键假设

### 逻辑实现助手 -> 测试助手

满足以下条件之一即可交接：

- 已生成 compile-safe 的逻辑骨架和 `logic-report.md`
- 或者已经明确列出关键 Stub、事务步骤和失败注入建议

最小完成标准：

- 高影响动作已有处理策略
- `logic-report.md` 已包含 `Critical Stubs`
- 若存在事务流程，已包含 `Transaction Steps`
- 若可推断失败注入点，已包含 `Failure Injection Points`

### 测试助手 -> 当前 feature 完成

满足以下条件之一即可视为测试阶段完成：

- 已生成最小可执行测试或测试计划
- 已生成 `QA-Dashboard.md`，且说明了覆盖情况、跳过项、警告与 `readinessScore`

最小完成标准：

- 已给出测试矩阵
- 已说明哪些测试被跳过以及原因
- 已明确当前是 `advisory` 还是 `blocking`

## 11. 最小交付原则

- 信息不足时，优先交付骨架、报告和明确假设。
- 优先保证规则正确、结构正确、可继续迭代。
- 优先保证 app 能打开、能运行、能审核，再逐步补齐逻辑和测试。
- 不为了“完整感”编造不存在的接口、模型或测试基础设施。

## 12. 变更回流规则

当页面实现、逻辑实现、测试验证或设计解析新增了以下任一信息时，必须先回流更新上游文档，再继续后续阶段：

- 当前 app 的页面范围、关键流程或交付边界发生变化
- 领域实体、状态、业务规则、高影响动作发生变化
- 接口契约、错误语义或本地仓储等价接口发生变化
- 交接判断、门禁判断、关键假设发生变化

统一回流顺序：

1. `docs/app.md`
2. `docs/domain.md`
3. `docs/api.md`
4. 当前阶段报告
5. skill

不要出现“代码已经改了，但文档和报告仍停留在旧规则”的状态。
