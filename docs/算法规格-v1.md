# 算法规格 v1

本文件定义当前 App 从“本地模板型推演”升级到“传统命理级严谨算法”的目标规格。

它不直接约束 UI 排版，只约束：

- 输入标准化
- 历法与天文换算
- 核心命理规则
- 四个功能的结构化输出
- 输出模板变量

## 1. 目标

- 在保持本地离线可运行的前提下，将今日、八字、合婚、取名四个功能升级为可复现、可解释、同输入稳定输出的传统命理规则引擎。
- 将“算法层”与“文案层”彻底拆开：
  - 算法层只输出结构化结果
  - 文案层只负责按页面模板组装说明
- 保留“娱乐参考 / 辅助参考”的产品表达，不伪装成专业保证性结论。

## 2. 参考来源

本轮优先采用以下三类参考：

### 2.1 历法与干支底座

- `sxtwl_cpp`
  - 作为寿星天文历的 C++ 实现，README 明确说明其依据天文历法算法实现，并支持真太阳时，适合命理研究。
  - 参考：<https://github.com/skydancep/sxtwl>

- `6tail/lunar`
  - 作为干支、节气、八字、五行、十神等的交叉校验参考。
  - README 明确写明：
    - 干支纪年支持以正月初一计
    - 支持以立春当日计
    - 支持以立春交接时刻计
    - 干支纪月支持以节当日计、节交接时刻计
  - 参考：<https://github.com/6tail/lunar-java>

### 2.2 真太阳时与太阳位置

- NOAA Solar Calculation Details
  - 用于真太阳时相关的太阳位置与均时差思路参考
  - 其页面说明太阳计算基于 Jean Meeus 的《Astronomical Algorithms》
  - 参考：<https://www.gml.noaa.gov/grad/solcalc/calcdetails.html>

- `Starainrt/astro`
  - 作为可选的天文算法实现参考，支持真太阳时、公农历转换、节气等
  - 参考：<https://github.com/Starainrt/astro>

## 3. 总体架构

算法按四层实现：

1. `Normalization`
   - 输入标准化
   - 经纬度、时区、历法、日期时间统一

2. `CalendarEngine`
   - 公历/农历/干支历转换
   - 节气、四柱、真太阳时

3. `DestinyEngine`
   - 五行、十神、旺衰、刑冲合害、格局、喜用神

4. `FeatureEngine`
   - 今日、八字、合婚、取名四个专项逻辑

5. `RenderAdapter`
   - 将结构化结果映射到页面输出模板

## 4. 输入标准化

### 4.1 输入模型

统一标准输入：

- `birthDate`
  - 公历日期
- `birthTime`
  - 本地时钟时间
- `calendarType`
  - `公历` / `农历`
- `gender`
  - `男` / `女`
- `location`
  - 出生地经度、纬度
  - MVP 若未提供城市库，则使用默认经度占位，并显式记录为假设
- `timezone`
  - 默认 `Asia/Shanghai`

附加输入：

- 合婚：
  - `personA`
  - `personB`
- 取名：
  - `surname`
  - 可选性别

### 4.2 标准化规则

- 统一先转为 `Gregorian + Local Time + Timezone`
- 若用户输入为农历：
  - 先转公历
  - 再进入节气与干支历流程
- 所有时间在进入四柱排盘前必须经过真太阳时修正

### 4.3 真太阳时修正

采用以下目标公式：

`真太阳时 = 平太阳时 + (当地经度 - 120°) * 4分钟 + 均时差(EOT)`

其中：

- 平太阳时：用户录入的本地标准时间
- 当地经度：出生地经度
- `120°`：东八区中央经线
- `EOT`：均时差

实现规则：

- 若已接入 `sxtwl` 或同等级天文底座，则优先使用底座输出
- 若自行实现：
  - 使用 Jean Meeus 体系下的太阳位置公式计算 `EOT`
- 真太阳时结果用于：
  - 时柱判定
  - 节气边界判定
  - 四柱最终确定

## 5. 历法引擎

### 5.1 目标输出

`CalendarEngine` 输出：

- `solarDateTime`
- `lunarDate`
- `jieQiContext`
  - 当前节气
  - 前后节气
  - 节气交接时刻
- `trueSolarDateTime`
- `fourPillars`
  - 年柱
  - 月柱
  - 日柱
  - 时柱

### 5.2 年月柱边界规则

- 年柱：
  - 默认按 `立春交接时刻` 判定
- 月柱：
  - 默认按 `节气交接时刻` 判定
- 日柱：
  - 按真太阳时所属日计算
- 时柱：
  - 按真太阳时所在时辰计算

### 5.3 可配置策略

为兼容不同流派，保留但不默认启用以下策略：

- 年柱按正月初一
- 年柱按立春当日
- 月柱按节当日

MVP 默认配置：

- `yearBoundaryMode = liChunInstant`
- `monthBoundaryMode = jieQiInstant`

## 6. 核心命理引擎

### 6.1 基础数据

固定规则表：

- 十天干
- 十二地支
- 五行映射
- 藏干表
- 地支刑冲合害破穿规则
- 天干合克规则
- 纳音表
- 十神映射表
- 长生十二运表

### 6.2 五行强弱打分

目标不是只做“出现次数统计”，而是做加权评分。

评分来源：

- 月令权重
- 透干权重
- 通根权重
- 同党 / 异党
- 生扶 / 克泄耗
- 季节旺衰

输出：

- `woodScore`
- `fireScore`
- `earthScore`
- `metalScore`
- `waterScore`
- `dayMasterStrength`
  - `极弱 / 弱 / 偏弱 / 中和 / 偏强 / 强 / 极强`

### 6.3 十神分析

围绕日主计算：

- 比肩
- 劫财
- 食神
- 伤官
- 偏财
- 正财
- 七杀
- 正官
- 偏印
- 正印

输出：

- 十神分布
- 主导十神
- 关键组合

### 6.4 格局识别

采用“候选格局 + 条件命中 + 优先级裁决”方式：

- 不直接用模糊文案判断
- 每个格局必须有明确命中条件

首批支持：

- 伤官配印
- 杀印相生
- 食神生财
- 官印相生
- 财官相生
- 身强财旺
- 身弱印比扶身

输出：

- `patternCandidates`
- `resolvedPattern`
- `patternConfidence`

### 6.5 喜用神判定

喜用神判定不直接按“缺什么补什么”。

规则顺序：

1. 判断日主强弱
2. 结合月令旺衰
3. 看全局寒暖燥湿
4. 判断扶抑方向
5. 输出喜神、用神、忌神、仇神

输出：

- `favorableElements`
- `supportingElements`
- `unfavorableElements`

## 7. 专项算法

### 7.1 今日

核心逻辑：

- 计算用户命局
- 计算今日干支
- 评估今日干支与命局喜用神关系
- 评估流日对原局的生扶、克泄耗、刑冲合害
- 产出今日参考结论

结构化输出：

- `dayStemBranch`
- `favorableLevel`
  - `低 / 平 / 中上 / 高`
- `supportReasons`
- `riskReasons`
- `yiTags`
- `jiTags`
- `oracleKey`

说明：

- `oracleKey` 只作为模板映射键
- 签语文案不参与核心计算

### 7.2 八字

核心逻辑：

- 四柱排盘
- 藏干展开
- 十神分布
- 五行加权打分
- 格局识别
- 喜用神判定

结构化输出：

- `fourPillars`
- `hiddenStems`
- `tenGodsSummary`
- `fiveElementScores`
- `dayMasterStrength`
- `resolvedPattern`
- `favorableElements`
- `unfavorableElements`

### 7.3 合婚

核心逻辑不采用单一“五行差值”。

评分维度：

- 双方日主关系
- 双方喜用神互补
- 双方夫妻宫关系
- 双方地支刑冲合害
- 双方五行结构互补度
- 双方强弱平衡度

建议权重：

- 喜用神互补：30
- 夫妻宫关系：20
- 地支关系：20
- 五行互补：20
- 日主关系：10

结构化输出：

- `compatibilityScore`
- `scoreBreakdown`
- `supportMatches`
- `conflictMatches`
- `marriagePalaceRelation`
- `overallBand`

### 7.4 取名

采用“约束满足 + 候选打分”而不是固定词库拼接。

硬约束：

- 满足喜用神补益方向
- 不触犯禁用字 / 生僻字 / 多音歧义严重字
- 姓名整体读音可接受

软约束：

- 五格剖象分
- 字义正向
- 语义搭配自然
- 音律顺口
- 重名风险较低

建议排序：

`总分 = 命理补益分 * 0.45 + 语义分 * 0.25 + 音律分 * 0.15 + 书写常用度 * 0.10 + 数理分 * 0.05`

说明：

- 五格剖象法仅作为次级权重，不作为唯一硬门槛
- NLP 语义过滤只做辅助，不替代命理约束

结构化输出：

- `namingNeed`
  - 需要增强的元素
- `candidateNames`
  - 名字
  - 五行补益说明
  - 语义说明
  - 音律说明
  - 数理说明
  - 总分

## 9. 工程实施顺序

按以下顺序推进：

1. `CalendarEngine`
   - 接入 `sxtwl` 或等价能力
   - 建立真太阳时与四柱排盘

2. `DestinyEngine`
   - 五行打分
   - 十神
   - 格局
   - 喜用神

3. `FeatureEngine`
   - 今日
   - 八字
   - 合婚
   - 取名

4. `RenderAdapter`
   - 适配现有页面输出

5. `Regression`
   - 为固定输入建立黄金样例
   - 保证同输入、同算法版本、同输出

## 10. 已知假设

- 当前 App 仍保留“娱乐参考 / 辅助参考”表达，不承诺唯一正确解释。
- 若出生地未填写，真太阳时将暂时使用默认经度，这会影响严谨度；该项应尽快补齐。
- 合婚与取名的部分流派规则存在分歧，本规格优先追求“规则明确、可复现、可解释”，而非兼容所有流派。

## 11. 当前不做

- 不在本规格里定义页面视觉稿
- 不在本规格里定义商业化策略
- 不在本规格里定义后端接口细节
- 不在本规格里承诺专家人工复核
