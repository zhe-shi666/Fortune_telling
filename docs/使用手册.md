# 使用手册

这个手册说明如何在 0 到 1 开发一个 iOS App 时，使用你的 3 个 skill 和 4 份约束文档协同工作。

适用对象：

- 你已经有这 3 个 skill：
  - `页面生成助手`
  - `逻辑实现助手`
  - `测试助手`
- 你已经有这 4 份文档：
  - `docs/rules.md`
  - `docs/app.md`
  - `docs/domain.md`
  - `docs/api.md`
- 你准备从一个想法开始，把 App 做到可运行、可迭代。

---

## 1. 这套体系的核心思想

你不是先搭一大堆项目结构，再让 AI 往里面填代码。

你真正的工作流是：

1. 先用文档把目标、边界、业务规则写清楚。
2. 尽快让 AI 生成一个可运行、可人工审核的 app 壳。
3. 再让 skill 按顺序生成页面、逻辑、测试。
4. 每轮迭代优先修正文档和规则中的歧义，再继续生成代码。

一句话理解：

- `rules.md` 管通用规则
- `app.md` 管当前 app 的目标和范围
- `domain.md` 管当前 app 的业务对象和规则
- `api.md` 管当前 app 的后端接口契约和错误语义
- 3 个 skill 负责把这些内容转成代码和报告

---

## 2. 每个文件分别负责什么

### `docs/rules.md`

这是通用规则文件。

它回答这些问题：

- 技术栈是什么
- 架构边界是什么
- 三个 skill 按什么顺序协作
- 哪些字段是共享协议
- 生成代码时哪些行为不允许

不要在这里写当前 app 的业务细节。

### `docs/app.md`

这是当前 app 的产品范围文件。

它回答这些问题：

- 这个 app 是干什么的
- 给谁用
- 当前版本只做哪些页面
- 哪些功能明确不做
- 关键用户流程是什么
- 当前 app 有哪些特有约束

### `docs/domain.md`

这是当前 app 的业务规则文件。

它回答这些问题：

- 核心实体有哪些
- 关键字段有哪些
- 状态有哪些
- 高影响动作有哪些
- 关键业务规则是什么
- 哪些地方还不稳定

### `docs/api.md`

这是当前 app 的接口契约文件。

它回答这些问题：

- MVP 需要哪些接口
- 每个接口的 `Method` / `Path` 是什么
- 请求和响应字段是什么
- 哪些错误语义会影响前端逻辑与测试
- 哪些接口契约仍然只是保守假设

### `页面生成助手`

负责把页面需求转成：

- 页面骨架
- 页面契约
- mock 数据
- UI 报告

### `逻辑实现助手`

负责把页面契约和业务规则转成：

- ViewModel
- 业务流程
- 依赖注入方案
- 逻辑报告

### `测试助手`

负责把页面和逻辑产物转成：

- 状态测试
- 行为测试
- 快照测试或测试计划
- QA 报告

## 3. 从 0 到 1 的推荐顺序

不要一上来就说“帮我做一个 app”。

更稳的顺序是下面这样。

### 第 1 步：先写 `app.md`

你先补这些内容：

- 一句话目标
- 目标用户
- MVP 范围
- 当前不做
- 页面清单
- 关键用户流程

你不需要一开始写得很完整，但至少要让 AI 知道：

- 这个 app 的核心目标是什么
- 第一版有哪些页面
- 哪些需求不做

建议你先把 `app.md` 填到能回答这 3 个问题：

1. 用户是谁
2. 用户第一版能完成什么
3. 什么暂时不做

### 第 2 步：再写 `domain.md`

你重点补这些内容：

- 核心实体
- 关键字段
- 状态枚举
- 关键业务规则
- 高影响动作
- 当前已知风险

如果你不知道要写多细，就记住一个标准：

只写那些会影响页面分支、逻辑判断、测试断言的内容。

### 第 3 步：补 `api.md`

在零人工或低人工模式下，接口文档不应该额外等你手写。

你要让 AI 先补出 `api.md` 的 v0，至少写清：

- MVP 需要的最小接口集合
- 每个接口的 `Method` / `Path`
- 最小 request / response 结构
- 会影响前端状态和测试的错误语义

如果真实后端还不存在，也允许先写保守假设，但必须写明是假设。

### 第 4 步：用页面生成助手做第一批页面

此时再让 AI 开始生成 UI。

你给页面生成助手的输入，应该至少包括：

- `docs/rules.md`
- `docs/app.md`
- 当前页面目标
- 相关设计稿、草图、截图或文字描述

如果项目从 0 开始，这一步不要只让 AI 生成页面文件，还要要求它生成最小可运行入口，让你可以立即打开 app 做人工审核。

### 第 5 步：用逻辑实现助手补业务流程

当页面骨架出来后，再补逻辑。

这一步的关键输入是：

- 页面契约
- `docs/domain.md`
- `docs/api.md`
- 页面报告
- 当前 app 的特殊业务要求

### 第 6 步：用测试助手补测试矩阵

测试不要等最后再想。

当页面和逻辑第一版出来后，就应该生成：

- 最小状态测试
- 行为测试
- 快照测试或替代计划
- QA 报告

## 4. 你在每个阶段应该怎么提问

下面是推荐问法。

### 阶段 A：定义 app

你可以这样发起：

```text
基于 docs/rules.md 的规则，帮我补全 docs/app.md。
这个 app 的目标是：……
目标用户是：……
第一版只做：……
明确不做：……
```

或者：

```text
这是一个从 0 到 1 的 iOS app。请先不要写代码，先帮我把 docs/app.md 填成第一版。
```

### 阶段 B：定义业务规则

你可以这样发起：

```text
基于 docs/app.md，帮我补全 docs/domain.md。
重点写核心实体、关键字段、状态枚举、高影响动作和关键业务规则。
```

### 阶段 C：定义接口契约

你可以这样发起：

```text
基于 docs/app.md 和 docs/domain.md，帮我补全 docs/api.md。
重点写 MVP 需要的最小接口清单、Method、Path、request/response 字段、错误语义和已知假设。
如果真实后端尚未确定，先给保守的 v0 契约，不要伪装成已确认事实。
```

### 阶段 D：开始做页面

你可以这样发起：

```text
使用 页面生成助手。
基于 docs/rules.md、docs/app.md 和下面的页面描述，生成 [某页面] 的第一版页面骨架。
如果信息不够，优先生成最小可编译版本，不要脑补复杂交互。
```

如果你有设计稿：

```text
使用 页面生成助手。
基于 docs/rules.md、docs/app.md 和这张设计稿，生成 [某页面]。
优先输出页面契约、主视图、mock 和 ui-layout-report。
```

### 阶段 E：补逻辑

你可以这样发起：

```text
使用 逻辑实现助手。
基于 docs/rules.md、docs/domain.md、docs/api.md、当前页面契约和 ui-layout-report，
补全这个页面的 ViewModel 和业务逻辑。
优先处理高影响动作，不要编造不存在的依赖基础设施。
```

### 阶段 F：补测试

你可以这样发起：

```text
使用 测试助手。
基于 docs/rules.md、docs/domain.md、docs/api.md、ui-layout-report 和 logic-report，
生成这个页面的最小测试矩阵、测试代码和 QA-Dashboard。
如果项目缺少快照框架或 in-memory 测试基建，先降级为测试计划。
```

## 5. 正确的迭代方式

这套体系最重要的一点是：

不要一发现结果不满意，就直接让 AI 重新生成一大坨代码。

更稳的顺序是：

1. 先看是 `rules.md` 的问题，还是 `app.md` 的问题，还是 `domain.md` 的问题
2. 先修文档
3. 再重新调用对应的 skill

举例：

### 情况 A：页面方向不对

优先检查：

- `app.md` 里的页面目标是不是没写清楚
- 是否明确了当前不做什么
- 是否给了足够的设计输入

不要一上来就骂页面 skill 生成错了。

### 情况 B：逻辑太脑补

优先检查：

- `domain.md` 的业务规则是不是太空
- 高影响动作是不是没定义
- 当前已知风险是不是没写

### 情况 C：测试太虚

优先检查：

- 逻辑报告里是不是没有失败注入点
- `domain.md` 是否没有写清哪些动作是高影响
- `app.md` 是否没有写清哪些页面必须 blocking

## 6. 什么时候该改文档，什么时候该改 skill

这是最容易混乱的地方。

### 改 `app.md`

当问题属于“当前这个 app 的目标或范围不清楚”时，改它。

例如：

- 页面到底有哪些
- 哪些功能这版不做
- 哪些页面必须走品牌风格

### 改 `domain.md`

当问题属于“业务规则不清楚”时，改它。

例如：

- 什么能创建
- 什么不能删除
- 哪些动作属于高影响
- 哪些失败要阻断

### 改 `rules.md`

当问题属于“跨项目通用规则不清楚”时，改它。

例如：

- 技术栈边界
- 架构模式
- skill 的协作顺序
- 共享字段协议

### 改 skill

当问题属于“AI 的工作流程本身不稳定”时，改 skill。

例如：

- 它没按顺序读取输入
- 它没有 fallback
- 它的报告模板和正文要求不一致
- 它和其他 skill 的共享协议对不上

一句话判断：

- 当前 app 的问题，改 `app.md` / `domain.md`
- 所有 app 都会反复遇到的问题，改 `rules.md` 或 skill

## 7. 最小实战模板

下面是一套最小的真实使用顺序。

### 第一次启动项目

1. 补 `docs/app.md`
2. 补 `docs/domain.md`
3. 选一个最核心页面
4. 用页面生成助手做第一页
5. 用逻辑实现助手补第一页逻辑
6. 用测试助手补第一页测试

不要同时铺 10 个页面。

### 第二轮迭代

1. 回看第一页报告
2. 修 `app.md` 或 `domain.md`
3. 再做第二个页面
4. 再补第二个页面逻辑和测试

### 第三轮以后

按 feature 逐步推进，而不是按文件类型推进。

正确方式：

- 做完一个 feature 的页面、逻辑、测试，再做下一个 feature

不推荐方式：

- 先做完所有页面，再回头补逻辑，再回头补测试

## 8. 常见错误

### 错误 1：`app.md` 太空就开始生成代码

结果：

- 页面方向漂
- 逻辑脑补严重
- 测试没有锚点

### 错误 2：把通用规则写进 `app.md`

结果：

- 同样的约束写两遍
- skill 和文档冲突时不知道听谁的

### 错误 3：把业务规则写进 `rules.md`

结果：

- 下一个 app 复用时污染规则

### 错误 4：页面、逻辑、测试分三个月做

结果：

- 前面生成的东西后面没人校正
- 逻辑和测试会脱节

### 错误 5：一出问题就重写 skill

结果：

- 真正的问题可能只是 `app.md` 太空

先判断问题属于：

- 规则问题
- 当前 app 需求问题
- 当前业务定义问题
- skill 工作流问题

再决定改哪里。

---

## 9. 你每次开一个新 app 时该做什么

每次新开一个 app，建议复制这 3 个文档模板：

- `docs/rules.md`
- `docs/app.md`
- `docs/domain.md`

然后：

1. 保留 `rules.md` 的大部分内容
2. 重写 `app.md`
3. 重写 `domain.md`
4. 沿用这 3 个 skill

也就是说：

- `rules.md` 偏通用资产
- `app.md` 和 `domain.md` 偏项目资产
- 3 个 skill 是工作流资产

---

## 10. 最后给你的实用建议

如果你想让这套体系真正高效，记住这 3 句话：

1. 先写清文档，再让 skill 生成代码。
2. 一次只推进一个 feature，不要同时铺太多页面。
3. 出问题先改文档和协议，再决定是否改 skill。

如果你遵守这三条，这套体系在中小型 iOS App 的 0 到 1 开发里会非常稳。
